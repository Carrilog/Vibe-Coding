# 核心模块详解

本文档逐一解析 `src/` 下每个核心模块的职责、关键函数、设计决策和实现细节。

---

## 1. index.ts — 主编排器

**文件**: `src/index.ts`
**职责**: 整个系统的入口和中枢，负责状态加载、消息循环、Agent 调用

### 关键函数

| 函数 | 职责 |
|------|------|
| `main()` | 初始化 DB、Channel、子系统，启动消息循环 |
| `loadState()` | 从 DB 加载路由状态、会话、已注册分组 |
| `startMessageLoop()` | 主轮询循环，每 2s 检查新消息 |
| `processGroupMessages()` | 处理单个分组的待处理消息 |
| `runAgent()` | 启动容器 Agent 并处理流式输出 |
| `recoverPendingMessages()` | 启动时恢复未处理的消息 |
| `getAvailableGroups()` | 获取所有可用分组 |

### 核心流程

```
main()
  ├── initDatabase()
  ├── loadState()
  ├── 初始化 Channels (connect + syncGroups)
  ├── startIpcWatcher()
  ├── startSchedulerLoop()
  ├── startSessionCleanup()
  ├── recoverPendingMessages()
  └── startMessageLoop()
        └── 每 2s:
            ├── getNewMessages() 从 DB 获取新消息
            ├── 按分组分发
            ├── Trigger 模式匹配（非 main 分组）
            └── GroupQueue.enqueueMessageCheck()
                └── runAgent()
                    ├── formatMessages() → XML
                    ├── runContainerAgent() → 启动容器
                    ├── 流式输出 → formatOutbound()
                    └── channel.sendMessage()
```

### 设计要点

- **状态恢复**: 如果 Agent 已发送部分输出后崩溃，cursor 会回滚，确保消息不丢失
- **Trigger 匹配**: 非 main 分组需要消息匹配 trigger pattern 才会触发 Agent
- **空闲超时**: 30 分钟无活动自动关闭容器
- **Stale Session 检测**: 通过正则匹配错误信息检测过期会话，自动清理重试

**职责**: 集中管理所有配置项，从环境变量和 .env 文件读取

### 关键配置项


## 3. db.ts — 数据库层

**文件**: `src/db.ts`
**职责**: SQLite 数据库的所有操作，包括 schema 管理、CRUD、迁移

### 数据库 Schema

```sql
-- 聊天元数据
CREATE TABLE chats (
  jid TEXT PRIMARY KEY,
  name TEXT,
  last_message_time TEXT,
  channel TEXT DEFAULT 'whatsapp',
  is_group INTEGER DEFAULT 0
);

-- 消息历史
CREATE TABLE messages (
  id TEXT PRIMARY KEY,
  chat_jid TEXT,
  sender TEXT,
  content TEXT,
  timestamp TEXT,
  is_bot_message INTEGER DEFAULT 0,
  reply_to_id TEXT,
  reply_to_sender TEXT,
  reply_to_content TEXT
);

-- 定时任务
CREATE TABLE scheduled_tasks (
  id TEXT PRIMARY KEY,
  group_folder TEXT,
  prompt TEXT,
  schedule_type TEXT,     -- 'cron' | 'interval' | 'once'
  schedule_value TEXT,
  context_mode TEXT DEFAULT 'isolated',
  script TEXT,
  next_run TEXT,
  last_run TEXT,
  status TEXT DEFAULT 'active',
  created_at TEXT
);

-- 任务执行日志
CREATE TABLE task_run_logs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  task_id TEXT,
  started_at TEXT,
  finished_at TEXT,
  duration_ms INTEGER,
  status TEXT,
  result TEXT,
  error TEXT
);

-- 路由状态 (KV 存储)
CREATE TABLE router_state (key TEXT PRIMARY KEY, value TEXT);

-- 会话映射
CREATE TABLE sessions (group_folder TEXT PRIMARY KEY, session_id TEXT);

-- 已注册分组
CREATE TABLE registered_groups (
  name TEXT PRIMARY KEY,
  folder TEXT UNIQUE,
  trigger_pattern TEXT,
  container_config TEXT,  -- JSON
  is_main INTEGER DEFAULT 0,
  channel TEXT DEFAULT 'whatsapp'
);
```

### 迁移策略

- 使用 `PRAGMA table_info()` 检查列是否存在
- 缺失列通过 `ALTER TABLE ADD COLUMN` 添加
- 支持从旧版 JSON 文件迁移到 SQLite

### 关键查询

```typescript
getNewMessages(since: string)
// 获取 since 之后的所有非 bot 消息，按时间排序

getMessagesSince(groupJid: string, since: string, limit: number)
// 获取特定分组的消息，带数量限制

getDueTasks()
// 获取 next_run <= now 且 status = 'active' 的任务
```

## 5. container-runner.ts — 容器执行

**文件**: `src/container-runner.ts`
**职责**: 构建容器参数、启动容器、处理流式输出

### 挂载策略

```
Main 分组:
├── 项目根目录 → /workspace/project (只读)
├── store/ → /workspace/store (读写)
├── groups/{folder}/ → /workspace/group (读写)
├── groups/global/ → /workspace/global (只读)
├── .claude/ → /workspace/.claude (读写)
├── ipc/ → /workspace/ipc (读写)
└── 额外挂载 → /workspace/extra/{path} (按 allowlist)

非 Main 分组:
├── groups/{folder}/ → /workspace/group (读写)
├── groups/global/ → /workspace/global (只读)
├── .claude/ → /workspace/.claude (读写)
├── ipc/ → /workspace/ipc (读写)
└── 额外挂载 → /workspace/extra/{path} (按 allowlist)
```

**关键区别**: 非 main 分组无法访问项目根目录和 store/

### 输出解析

使用哨兵标记分隔输出：

```
---NANOCLAW_OUTPUT_START---
Agent 的实际响应内容
---NANOCLAW_OUTPUT_END---
```

### 超时管理

- 硬超时: `max(CONTAINER_TIMEOUT, IDLE_TIMEOUT + 30s)`
- 流式输出时重置超时计时器
- 超时后优雅停止容器

### OneCLI 集成

```typescript
// 通过 OneCLI SDK 获取凭证注入配置

// 配置注入到容器环境变量中
```

## 7. group-queue.ts — 分组队列

**文件**: `src/group-queue.ts`
**职责**: 管理每个分组的容器执行队列，控制全局并发

### 状态机

```
每个分组的状态:
  IDLE → ACTIVE → IDLE_WAITING → ACTIVE (收到新消息)
                                → IDLE (超时)

全局状态:
  活跃容器数 ≤ MAX_CONCURRENT_CONTAINERS (5)
  超出的排队等待
```

### 关键方法

```typescript
enqueueMessageCheck(group, callback)
// 如果容器空闲等待中 → 直接发 IPC 消息
// 如果容器活跃 → 排队
// 如果无容器 → 启动新容器

enqueueTask(group, task, callback)
// 任务优先级高于消息
// 防止同一任务重复入队

sendMessage(group, content)
// 写入 IPC input 文件，容器轮询读取

closeStdin(group)
// 写入关闭信号，容器开始收尾

registerProcess(group, process)
// 注册活跃容器进程

notifyIdle(group)
// 容器进入空闲等待状态
// 如果有待处理任务 → 抢占

shutdown()
// 优雅关闭：分离容器（不杀死），等待排空
```

### 重试策略

```
重试间隔 = 5s × 2^(retryCount - 1)
最大重试次数 = 5

第1次: 5s
第2次: 10s
第3次: 20s
第4次: 40s
第5次: 80s
```

## 8. env.ts — 环境变量解析

**文件**: `src/env.ts`
**职责**: 安全地解析 .env 文件

```typescript
readEnvFile(keys: string[]): Record<string, string | undefined>
// 只返回请求的 key，不加载到 process.env
// 防止密钥泄露到子进程
```

**设计决策**: 传统的 `dotenv.config()` 会将所有变量加载到 `process.env`，子进程会继承。NanoClaw 选择性读取，确保密钥不会进入容器环境。

## 11. logger.ts — 结构化日志

**文件**: `src/logger.ts`
**职责**: 带颜色、时间戳、PID 的结构化日志

### 日志级别

| 级别 | 数值 | 颜色 | 输出 |
|------|------|------|------|
| debug | 20 | 蓝色 | stdout |
| info | 30 | 绿色 | stdout |
| warn | 40 | 黄色 | stderr |
| error | 50 | 红色 | stderr |
| fatal | 60 | 红底 | stderr |

### 特性

- `LOG_LEVEL` 环境变量控制阈值
- 结构化数据以 key=value 形式输出
- Error 对象自动提取 stack trace
- 注册 `uncaughtException` 和 `unhandledRejection` 处理器
