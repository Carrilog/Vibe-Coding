# 系统架构

## 架构总览

NanoClaw 采用**单进程编排器 + 多容器 Agent** 架构。宿主机上运行一个 Node.js 进程负责消息路由、任务调度和状态管理；每个对话组的 AI Agent 在独立的 Docker 容器中执行。

## 核心架构决策

### 1. 单进程编排

所有编排逻辑在一个 Node.js 进程中运行，没有微服务拆分。

**优点**：
- 简单，易于理解和调试
- 无需服务发现、RPC 框架
- SQLite 同步 API 天然适合单进程

**代价**：
- 单点故障（通过 launchd KeepAlive 缓解）
- 垂直扩展受限（个人用户场景足够）

### 2. 容器隔离（而非权限检查）

Agent 运行在 Docker 容器中，通过 OS 级隔离保证安全，而不是在应用层做权限检查。

```
┌─────────────────────────────────────────┐
│              容器边界                     │
│                                         │
│  可访问:                                 │
│  ├── /workspace/group/    (自己的目录)   │
│  ├── /workspace/global/   (全局只读)     │
│  ├── /workspace/ipc/      (IPC 通道)     │
│  ├── /workspace/extra/    (额外挂载)     │
│  └── /workspace/.claude/  (Claude 配置)  │
│                                         │
│  不可访问:                               │
│  ├── 宿主机文件系统                      │
│  ├── 其他容器的数据                      │
│  ├── .env / 密钥文件                     │
│  └── 网络（受限）                        │
└─────────────────────────────────────────┘
```

### 3. 文件系统 IPC

容器与宿主机通过文件系统进行通信，不使用网络 socket 或 gRPC。

```
/workspace/ipc/
├── messages/          # 容器 → 宿主机：发送消息
├── tasks/             # 容器 → 宿主机：任务操作
├── errors/            # 失败的 IPC 文件
├── input/             # 宿主机 → 容器：后续消息输入
├── tasks-snapshot.json # 宿主机 → 容器：当前任务列表
└── groups-snapshot.json# 宿主机 → 容器：可用分组列表
```

### 4. Channel 注册模式

渠道通过工厂模式自注册，启动时自动加载：

```typescript
// channels/registry.ts — 注册表
registerChannel('telegram', telegramFactory)

// channels/index.ts — 自注册入口
import './telegram'  // 副作用导入触发注册
```

### 5. Skills as Branches

功能扩展以 git branch 形式存在，而非核心代码：

```
main branch          ← 核心功能
├── skill/whatsapp   ← WhatsApp 渠道
├── skill/telegram   ← Telegram 渠道
├── skill/slack      ← Slack 渠道
├── skill/voice      ← 语音转写
└── skill/pdf        ← PDF 阅读
```

## 进程模型

```
NanoClaw 主进程
│
├── 消息循环 (Message Loop)
│   └── 每 2 秒轮询 SQLite 获取新消息
│
├── 任务调度器 (Task Scheduler)
│   └── 每 60 秒检查到期任务
│
├── IPC 监听器 (IPC Watcher)
│   └── 每 1 秒扫描 IPC 目录
│
├── 会话清理 (Session Cleanup)
│   └── 每 24 小时清理过期会话
│
├── 远程控制 (Remote Control) [可选]
│   └── Claude Code 远程调试会话
│
└── Channel 连接
    ├── WhatsApp WebSocket
    ├── Telegram Bot API
    ├── Slack Socket Mode
    └── Discord Gateway
```

## 并发模型

```
GroupQueue 管理并发:

┌─────────────────────────────────────────────┐
│  全局并发限制: MAX_CONCURRENT_CONTAINERS = 5 │
│                                             │
│  Group-main:  [ACTIVE] ← 容器运行中         │
│  Group-A:     [IDLE]   ← 容器空闲等待 IPC   │
│  Group-B:     [QUEUED] ← 等待容器槽位        │
│  Group-C:     [QUEUED] ← 等待容器槽位        │
│                                             │
│  重试策略: 指数退避 5s × 2^(n-1), 最多 5 次  │
└─────────────────────────────────────────────┘
```

## 数据存储架构

```
SQLite (data/nanoclaw.db)
├── chats          — 聊天元数据 (jid, name, channel, is_group)
├── messages       — 消息历史 (sender, content, timestamp, reply)
├── scheduled_tasks— 定时任务 (cron/interval/once, prompt, group)
├── task_run_logs  — 任务执行日志
├── router_state   — 路由状态 (最后处理时间戳)
├── sessions       — 分组会话 ID 映射
└── registered_groups — 分组注册信息 (trigger, config)

文件系统
├── groups/{name}/     — 分组数据目录
│   ├── CLAUDE.md      — 分组记忆
│   ├── logs/          — 容器日志
│   └── ipc/           — IPC 通道
├── groups/global/     — 全局共享记忆
└── data/              — 运行时数据
```
