# 安全模型

## 信任模型

NanoClaw 定义了四个信任层级：

```
┌─────────────────────────────────────────────────────────┐
│  信任层级                                                │
│                                                         │
│  ┌─────────────────┐                                    │
│  │  宿主机进程       │  ← 完全信任                       │
│  │  (NanoClaw 编排器)│     可访问所有资源                 │
│  └────────┬────────┘                                    │
│           │                                             │
│  ┌────────▼────────┐                                    │
│  │  Main 分组       │  ← 高信任                          │
│  │  (管理员自聊)     │     私人对话，可注册分组/管理任务    │
│  └────────┬────────┘                                    │
│           │                                             │
│  ┌────────▼────────┐                                    │
│  │  非 Main 分组    │  ← 低信任                          │
│  │  (其他用户群组)   │     可能包含恶意用户                │
│  └────────┬────────┘                                    │
│           │                                             │
│  ┌────────▼────────┐                                    │
│  │  容器 Agent      │  ← 沙箱                            │
│  │  (Docker 隔离)   │     OS 级隔离，受限文件系统         │
│  └─────────────────┘                                    │
│                                                         │
│  ┌─────────────────┐                                    │
│  │  用户消息         │  ← 不信任                          │
│  │  (外部输入)       │     潜在 prompt injection          │
│  └─────────────────┘                                    │
└─────────────────────────────────────────────────────────┘
```

## 权限矩阵

| 能力 | Main 分组 | 非 Main 分组 |
|------|-----------|-------------|
| 访问项目根目录 | 只读 | 不可 |
| 访问 store/ | 读写 | 不可 |
| 访问自己的 group/ | 读写 | 读写 |
| 访问 global/ | 只读 | 只读 |
| 注册新分组 | 可以 | 不可 |
| 刷新分组元数据 | 可以 | 不可 |
| 创建定时任务 | 可以 | 仅自己分组 |
| 管理其他分组任务 | 可以 | 不可 |
| 额外挂载 | 按 allowlist | 按 allowlist（可强制只读） |

## 安全边界详解

### 1. 容器隔离（主要安全边界）

```
宿主机文件系统
├── /Users/...          ← 容器不可见
├── /etc/...            ← 容器不可见
├── ~/.ssh/             ← 容器不可见
├── ~/.aws/             ← 容器不可见
└── nanoclaw/
    ├── .env            ← 容器不可见（密钥在此）
    ├── src/            ← 容器不可见（除 main 只读）
    └── groups/
        ├── main/       ← 仅 main 容器可见
        ├── group-a/    ← 仅 group-a 容器可见
        └── global/     ← 所有容器只读
```

**设计哲学**: 不在应用层做权限检查，而是让 OS 级容器隔离来保证安全。Agent 在容器内拥有完整的文件系统操作能力，但只能看到挂载进去的目录。

### 2. 挂载安全 (mount-security.ts)

额外挂载通过外部 allowlist 验证：

```json
// ~/.config/nanoclaw/mount-allowlist.json
// 存储在项目外部，容器无法篡改
{
  "allowedRoots": [
    {
      "hostPath": "/Users/me/projects",
      "readWrite": true
    }
  ],
  "blockedPatterns": [
    ".ssh", ".gnupg", ".aws", ".azure", ".kube",
    ".docker", "credentials", ".env", ".netrc",
    ".npmrc", ".pypirc", "id_rsa", "id_ed25519",
    "private_key", ".secret"
  ],
  "nonMainReadOnly": true
}
```

验证流程：
```
请求挂载 /Users/me/projects/data
    │
    ├── 1. 路径在 allowedRoots 内？ → 否则拒绝
    ├── 2. 路径匹配 blockedPatterns？ → 是则拒绝
    ├── 3. 解析符号链接 → 防止绕过
    ├── 4. 容器路径合法？ → 无 .. / 无绝对路径 / 无冒号
    ├── 5. 非 main 分组 + nonMainReadOnly？ → 强制只读
    └── 6. 通过 → 挂载到 /workspace/extra/{path}
```

### 3. 凭证隔离 (OneCLI Agent Vault)

```
传统方式（不安全）:
  .env → process.env → 容器环境变量 → Agent 可读取密钥

NanoClaw 方式（安全）:
  .env → readEnvFile() → 仅宿主机进程持有
  容器 → API 请求 → OneCLI Gateway → 注入凭证 → Claude API
  
  容器内无任何 API Key
```

### 4. IPC 授权

```
容器写入 IPC 文件时:
  ├── 文件路径包含分组标识
  ├── 宿主机通过路径确定来源分组
  ├── 验证操作权限:
  │   ├── register_group → 仅 main
  │   ├── refresh_groups → 仅 main
  │   ├── schedule_task → main 可操作所有，非 main 仅自己
  │   └── 其他任务操作 → 同上
  └── 失败的 IPC 文件移到 errors/
```

### 5. 发送者白名单 (sender-allowlist.ts)

```json
// ~/.config/nanoclaw/sender-allowlist.json
{
  "default": {
    "allow": "*",
    "mode": "trigger"
  },
  "chats": {
    "group-123@g.us": {
      "allow": ["alice@s.whatsapp.net", "bob@s.whatsapp.net"],
      "mode": "trigger"
    },
    "spam-group@g.us": {
      "allow": [],
      "mode": "drop"
    }
  },
  "logDenied": true
}
```

两种模式：
- **trigger**: 非白名单用户的消息存储但不触发 Agent
- **drop**: 非白名单用户的消息直接丢弃

### 6. 会话隔离

每个分组有独立的 Claude 会话：
- 会话 ID 存储在 SQLite `sessions` 表
- 不同分组的对话历史完全隔离
- 分组 A 的用户无法看到分组 B 的对话

### 7. 路径安全 (group-folder.ts)

```
输入验证:
  ├── 只允许: [a-zA-Z0-9_-]
  ├── 禁止: .. / 路径分隔符
  ├── 禁止: 保留名 "global"
  └── 解析后检查: 必须在 GROUPS_DIR 内
```

## 安全威胁与缓解

| 威胁 | 缓解措施 |
|------|----------|
| Prompt Injection | 消息 XML 转义；容器隔离限制影响范围 |
| 容器逃逸 | Docker 隔离；非 root 用户；最小挂载 |
| 密钥泄露 | OneCLI 网关；.env 不加载到 process.env |
| 路径穿越 | 多层验证；符号链接解析；bounds check |
| 跨分组访问 | 独立挂载命名空间；IPC 授权检查 |
| 敏感文件挂载 | blockedPatterns；外部 allowlist |
| 恶意 IPC | 来源验证；权限检查；错误隔离 |
| 服务拒绝 | 容器超时；并发限制；指数退避 |
