# HelloAgents v0.1 — Core 模块开发记录

> 版本：v0.1 | 日期：2026-04-18 | 目标：最小化可运行的 Agent Core

## 1. 版本概述

本版本实现了 HelloAgents 框架的 `core/` 模块最小化版本，目标是：

- 保持与原项目完全一致的文件结构和命名
- 仅保留核心功能，移除高级特性（异步、多 Provider、子 Agent 等）
- 能够通过本地 Ollama（`deepseek-r1:32b`）实际运行
- 作为后续逐步迭代学习的基础骨架

已验证：导入正常，Ollama 调用成功（延迟 ~33s，82 tokens）。

## 3. 架构总览

### 核心数据流

```
用户输入
  ↓
Agent._build_messages()     构建消息列表（system prompt + history + user input）
  ↓
HelloAgentsLLM.invoke()     Facade 层，合并默认参数
  ↓
create_adapter()            工厂函数，选择适配器
  ↓
OpenAIAdapter.invoke()      实际 HTTP 调用（OpenAI SDK）
  ↓
LLMResponse                 标准化响应对象
```

### 工具调用流程

```
Agent._build_tool_schemas()       从 ToolRegistry 构建 JSON Schema
  ↓
HelloAgentsLLM.invoke_with_tools()  带工具定义的 LLM 调用
  ↓
LLMToolResponse                    包含 tool_calls 列表
  ↓
Agent._execute_tool_call()         执行工具 → 返回字符串结果
```

### 设计模式一览

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| 适配器模式 | `BaseLLMAdapter` / `OpenAIAdapter` | 统一不同 LLM Provider 的接口 |
| 外观模式 | `HelloAgentsLLM` | 隐藏适配器细节，提供简洁 API |
| 工厂模式 | `create_adapter()` | 根据配置自动选择适配器实现 |
| 模板方法 | `Agent(ABC)` | 基类提供通用能力，子类实现 `run()` |
| 值对象 | `Message`, `LLMResponse`, `ToolCall` 等 | 不可变数据传输对象 |

## 4. 各文件详解

### 4.1 `exceptions.py` — 异常体系

**状态：✅ 完整实现**（与原项目一致）

```
HelloAgentsException          框架基础异常
├── LLMException              LLM 调用相关
├── AgentException            Agent 执行相关
├── ConfigException           配置相关
└── ToolException             工具相关
```

作用：提供分层异常类型，便于上层按类别捕获和处理错误。当前主要使用 `HelloAgentsException`，子类为后续细粒度错误处理预留。

### 4.4 `llm_response.py` — LLM 响应对象

**状态：✅ 完整实现**（与原项目一致）

四个 dataclass，构成 LLM 层的标准化输出：

| 类 | 用途 | 关键字段 |
|----|------|---------|
| `ToolCall` | 单次工具调用 | `id`, `name`, `arguments` |
| `LLMToolResponse` | 工具调用响应 | `content`, `tool_calls`, `model`, `usage`, `latency_ms` |
| `LLMResponse` | 标准响应 | `content`, `model`, `usage`, `latency_ms`, `reasoning_content` |
| `StreamStats` | 流式统计 | `model`, `usage`, `latency_ms`, `reasoning_content` |

**`LLMResponse` 关键方法：**

- `__str__()` — 直接返回 content
- `__repr__()` — 调试信息（model、latency、tokens、content_length）
- `to_dict()` — 序列化（条件包含 reasoning_content）

### 4.5 `llm_adapters.py` — LLM 适配器层

**状态：✅ 完整实现**（仅保留 OpenAI 适配器）

这是框架与 LLM Provider 之间的桥梁，采用适配器模式。

**抽象基类：`BaseLLMAdapter(ABC)`**

| 方法 | 类型 | 说明 |
|------|------|------|
| `__init__(api_key, base_url, timeout, model)` | 构造器 | 初始化连接参数，`_client=None` 延迟创建 |
| `create_client()` | 抽象 | 创建 Provider 客户端实例 |
| `invoke(messages, **kwargs)` | 抽象 | 非流式调用 → `LLMResponse` |
| `stream_invoke(messages, **kwargs)` | 抽象 | 流式调用 → `Iterator[str]` |
| `invoke_with_tools(messages, tools, **kwargs)` | 抽象 | 工具调用 → `LLMToolResponse` |
| `_is_thinking_model(model_name)` | 具体 | 启发式判断是否为推理模型（检测 reasoner/o1/o3/thinking） |

**具体实现：`OpenAIAdapter(BaseLLMAdapter)`**

| 方法 | 核心逻辑 |
|------|---------|
| `create_client()` | 延迟导入 `openai.OpenAI`，传入 api_key/base_url/timeout |
| `invoke()` | `chat.completions.create` → 提取 content/reasoning_content/usage → `LLMResponse` |
| `stream_invoke()` | `stream=True` → yield delta.content → 完成后存储 `self.last_stats` |
| `invoke_with_tools()` | 传入 tools/tool_choice → 解析 tool_calls → `LLMToolResponse` |

**工厂函数：`create_adapter(api_key, base_url, timeout, model)`**

当前始终返回 `OpenAIAdapter`。后续可根据 base_url 或 provider 参数分发到 `AnthropicAdapter`、`GeminiAdapter` 等。

**与原项目差异：** 移除了 `AnthropicAdapter`、`GeminiAdapter`、异步方法（`create_async_client`、`astream_invoke`）。

### 4.6 `llm.py` — LLM 统一接口（Facade）

**状态：✅ 完整实现**（简化版）

**类：`HelloAgentsLLM`** — 框架对外的 LLM 入口

**构造器参数与默认值：**

| 参数 | 环境变量 | 默认值 |
|------|---------|--------|
| `model` | `LLM_MODEL_ID` | 无（必填） |
| `api_key` | `LLM_API_KEY` | `"ollama"` |
| `base_url` | `LLM_BASE_URL` | `"http://localhost:11434/v1"` |
| `timeout` | `LLM_TIMEOUT` | `60` |
| `temperature` | — | `0.7` |
| `max_tokens` | — | `None` |

**关键方法：**

| 方法 | 说明 |
|------|------|
| `think(messages, temperature)` | 流式调用 + 控制台打印（带 emoji 状态提示），yield 文本块 |
| `invoke(messages, **kwargs)` | 非流式调用，合并默认参数后委托适配器 |
| `stream_invoke(messages, **kwargs)` | 纯流式调用（无打印副作用），yield 文本块 |
| `invoke_with_tools(messages, tools, tool_choice, **kwargs)` | 工具调用，合并参数后委托适配器 |

`think()` vs `stream_invoke()`：`think()` 额外打印状态信息到 stdout，适合 CLI 交互；`stream_invoke()` 是纯 API，适合程序集成。

**与原项目差异：** 移除了所有异步方法（`ainvoke`、`astream_invoke`、`ainvoke_with_tools`）。默认指向 Ollama 本地服务。

### 4.7 `agent.py` — Agent 抽象基类

**状态：✅ 完整实现**（简化版）

**类：`Agent(ABC)`** — 所有 Agent 的基类

**构造器：** `__init__(name, llm, system_prompt, config, tool_registry)`

| 属性 | 说明 |
|------|------|
| `name` | Agent 名称 |
| `llm` | `HelloAgentsLLM` 实例 |
| `system_prompt` | 系统提示词 |
| `config` | `Config` 实例 |
| `tool_registry` | 工具注册表（可选） |
| `_history` | `List[Message]` 简单列表 |

**方法总览：**

| 方法 | 类型 | 说明 |
|------|------|------|
| `run(input_text, **kwargs)` | 抽象 | 子类必须实现的执行入口 |
| `add_message(message)` | 历史管理 | 追加消息 |
| `clear_history()` | 历史管理 | 清空历史 |
| `get_history()` | 历史管理 | 返回历史副本 |
| `_build_messages(input_text)` | 消息构建 | 组装 system + history + user 消息列表 |
| `_build_tool_schemas()` | 工具能力 | 从 registry 构建 OpenAI Function Calling JSON Schema |
| `_map_parameter_type(param_type)` | 工具能力 | 参数类型映射到 JSON Schema 类型（静态方法） |
| `_convert_parameter_types(tool_name, param_dict)` | 工具能力 | 将 LLM 返回的字符串参数转换为声明的类型 |
| `_execute_tool_call(tool_name, arguments)` | 工具能力 | 执行工具调用，返回字符串结果 |

**`_build_tool_schemas()` 支持两种工具模型：**
1. 结构化工具对象（通过 `tool.get_parameters()` 获取参数元数据）
2. 简单函数注册（通过 `registry._functions`，统一为 `input: string` 单参数）

**与原项目差异：** 历史管理从 `HistoryManager` 简化为普通列表。移除了 `TokenCounter`、`TraceLogger`、`SkillLoader`、`SessionStore`、压缩、子 Agent、异步生命周期等。`_execute_tool_call` 简化为直接调用 `tool.run()` 而非 `tool.run_with_timing()`。
